# Review: Codex follow-up on v0.4.4 conformance suite - 2026-07-05

## Verdict

v0.4.4 is a strong additive release: CI exists, vector regeneration is fresh,
anchors verify, and the new machine-readable conformance suite closes a real
multi-language implementation gap.

However, the review found two consensus-relevant edges that are now partly
encoded by the oracle/vectors and should be resolved before downstream
implementations treat `vectors.json` as complete law.

This note is written as a handoff for Claude/Sonnet to adjudicate and patch.

## Findings

### P1 - ATP budget enforcement performs a step before checking budget

Location: `impl/sigma_glyph.py`, `eval_hash()`.

Current loop shape:

```python
nt = step(t, store, stats, limits)
...
spent += 1
t = nt
if spent >= atp and step(t, store, stats, limits) is not None:
    return ("dis", R_ATP), spent
```

Observed behavior:

```text
eval(I K, atp=0) -> DISSONANCE(ATP Exhausted), atp_spent=1
eval(I K, atp=1) -> K, atp_spent=1
```

The first result is now encoded in `tests/spec_conformance/vectors.json`:
`EV-TV4-IK-ATP0` expects `atp_spent: 1` even though the supplied budget is 0.
`EV-TV7-OMEGA-0` does the same.

Why this matters:

- Book I says each R-I/R-K/R-S/R-R firing costs exactly 1 ATP and `eval(term,
  atp: uint32)` is total over the canonical outcomes. A budget of 0 should not
  be able to spend 1 unless the spec explicitly defines "spent" as attempted
  work after exhaustion, which it currently does not.
- The lookahead `step(t, ...)` is not pure. For `REF`, it may resolve storage
  and raise `Unresolved`.

Manual probe:

```python
# term = APPLY(I, REF(missing))
eval(term, atp=1) -> raises raw Unresolved
eval(term, atp=2) -> DISSONANCE(Unresolved Reference), atp_spent=1
```

That raw exception violates the intended totalization contract.

Concrete resolution options:

1. Prefer: enforce budget before firing the next redex. If `spent == atp` and
   the current term is not normal, return `DISSONANCE(ATP Exhausted)` with
   `atp_spent == spent`. This makes `eval(I K, 0)` spend 0.
2. If the current behavior is intentional, Book I must explicitly define
   over-budget attempted work and permit `atp_spent > atp`, then fix the
   lookahead so it cannot leak raw `Unresolved`.

Suggested new vectors:

- `EV-TV4-IK-ATP0` should assert the chosen `atp_spent` semantics explicitly.
- `EV-EXACT-BUDGET-THEN-UNRESOLVED-REF`: `APPLY(I, REF(missing))` with
  exact budget 1 must return a canonical outcome, not raise.

### P1 - Missing dead arguments are not pinned by vectors and may diverge

Location: `impl/sigma_glyph.py`, `load()`.

Current `load()` eagerly materializes both APPLY children:

```python
l = load(n["left"], store, stats, limits)
r = load(n["right"], store, stats, limits)
if l is None or r is None: return None
return ("app", l, r)
```

Manual probe:

```text
term = APPLY(APPLY(K, I), missing)
current oracle -> DISSONANCE(Unresolved Reference), atp_spent=0
```

A lazy normal-order implementation could instead recognize the root R-K redex
and reduce to `I` without resolving the unused `y`. The prose rule
`APPLY(APPLY(K, x), y) -> x` plus "normal order, leftmost-outermost" does not
clearly say that all child hashes must resolve before a root redex may fire.

Why this matters:

This is exactly the kind of gap where two independently reasonable
implementations can disagree on the result hash.

Concrete resolution options:

1. If eager materialization is normative, Book I §3.5 should say that resolving
   an APPLY node recursively resolves both children before redex recognition.
   Add a vector for `K` with a missing dead argument.
2. If normal-order laziness is normative, change the oracle representation so
   APPLY children can remain hashes/thunks until demanded, and add the inverse
   vector: `APPLY(APPLY(K, I), missing) -> I`.

I recommend deciding this explicitly before external implementations claim
v0.4.4 conformance.

### P2 - Version metadata is stale/confusing

Locations:

- `README.md` says `Current: v0.4.0`.
- `CHANGELOG.md` and `spec/ANCHORS.txt` advertise v0.4.4.
- `tests/spec_conformance/generate.py` emits `SPEC_VERSION = "0.4.3"`.
- `tests/spec_conformance/vectors.json` therefore reports
  `"spec_version": "0.4.3"`.

This may be intentional because Book I text did not change in v0.4.4, but it
creates an avoidable reporting ambiguity: should downstream implementations
claim "Book I 0.4.3 conformance" or "suite v0.4.4 conformance"?

Concrete text proposal:

- Update README to `Current: v0.4.4`.
- In `tests/spec_conformance/README.md`, distinguish:
  - `spec_version`: Book I document version whose anchor is tested.
  - `suite_version` or release version: conformance-suite package version.
- Optionally add `"suite_version": "0.4.4"` to `vectors.json` while keeping
  `"spec_version": "0.4.3"` if that is the intended anchor semantics.

## Suggested patch order

1. Decide ATP exhaustion semantics first; regenerate vectors afterward.
2. Decide eager-vs-lazy APPLY child resolution; add the missing conformance
   vector in the same patch.
3. Update README/version metadata and conformance README wording.
4. Rerun full CI plus `tools/verify_anchors.py`; refresh `ANCHORS.txt` if any
   anchored artifact changes.
