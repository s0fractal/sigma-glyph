# Review: ADR-007 design gate (proposals/ADR-007-governed-anchors.md, PROPOSED)

Reviewer: independent adversarial pass-2 (no code execution). Primary sources read: ADR-007 text and schemas; Book III §2 precedent; tools/anchor_governance.py.

## Verdict

Gate-worthy with P1s to close before adoption. The mechanism is sound and materially stronger than signed git tags alone, but the bootstrap story, jurisdiction binding, and the verifier need sharper scoping and a few MUST-level checks and algorithms added to prevent replay/misattribution. I ran no code; tool behaviors cited are from reading tools/anchor_governance.py and ADR-007.

## 1) Attack the bootstrap: 1-of-1 self-signed genesis vs signed git tags — exact delta or “ceremony”?

Not ceremony. The governed chain stops concrete attacks that signed tags cannot:

- Single-key compromise forward-writes history
  - Scenario: one maintainer key K signs v0.6.1 legitimately. Later K is stolen. Attacker signs a v0.6.2 git tag with K. Tag verification trusts K and accepts v0.6.2; plain tags have no machine-checkable revocation/rotation semantics.
  - Under ADR‑007 with a threshold policy G1 (e.g., min_sigs = 2 over N actors), an attacker with only K cannot satisfy an adoption warrant for the next anchor-set; they would need ≥2 distinct bound actors. “Revocation is a warrant” (Warrant §5.1) gives the legitimate maintainers a forward action to exclude K; post‑revocation, K’s warrants no longer count toward the threshold.

- Succession without out‑of‑band trust
  - Signed tags cannot express/verifiably adopt “policy changed; these are the new actors/quorum” in a way a tool can enforce.
  - ADR‑007 models such changes as warrants: a new governance profile adopted under the prior threshold makes a machine-checkable succession. A fresh verifier that trusts the governance profile + threshold lineage will accept vX+1 signed by founder+model‑C after model‑A retires, without social context.

- Replay of old signatures after key-state change
  - Plain tags remain “valid” forever. In ADR‑007, the active threshold and key-state at settlement time decide which signatures count. Once a deprecation or revocation warrant is settled (§5.1), signatures by that actor no longer contribute to meeting the threshold; verifiers that consider settlement-active state enforce this automatically.

Arithmetic example (why tags fail, governance succeeds):
- Actors = {founder, A, B}, min_sigs = 2.
- Attacker steals A. With tags: A alone can mint v0.6.2 — accepted by any client that trusts A’s key. Under governance: A alone cannot reach 2; even A+attacker cannot forge B.

Conclusion: The “difference class” vs signed tags is (a) threshold authorization and (b) revocation/rotation and succession as warrants under an authoritative, machine-checkable policy. Those stop real attacks; this is not ceremony.

## 2) Attack the ancestor chain across jurisdictions: can a fork replay the public anchor-set chain to a fresh verifier; do we need a jurisdiction root like Book III §2?

Replay risk exists unless the jurisdiction is pinned and enforced.

- Observation from sources: the anchor‑set blob schema (as drafted) carries release, optional ancestor, and anchors, but no jurisdiction field; the same blob bytes can live in multiple stores.
- Attack: copy a legitimate anchor‑set blob and its adoption warrant into a different store; present them to a fresh verifier. If the verifier has not pinned a jurisdiction root (Warrant §9), it cannot tell “original” from “replayed.” Even if a verifier pins roots globally, naive tools that cache blobs by hash without comparing the anchor‑set’s jurisdiction to the adoption’s root can misattribute across stores.

Mitigation: mirror Book III’s jurisdiction binding.

P1 (spec silent): ADR‑007 must normatively bind anchor‑sets to a jurisdiction and require verifiers to enforce settlement-active roots.

- Schema addition (anchor‑set blob):
  ```
  {
    "governance": "sigma-glyph.anchor-set@v1",
    "jurisdiction": "<hex64: WarrantID of the settlement-active genesis root for this jurisdiction>",
    "release": "vX.Y.Z",
    "ancestor": "<hex64, optional>",
    "anchors": [ ... ]
  }
  ```
  - MUST: A genesis anchor‑set MUST carry a jurisdiction and MUST omit ancestor.
  - MUST: A successor anchor‑set MUST carry an ancestor equal to the prior adopted anchor‑set in this jurisdiction (see §3 and P1‑C).

- Verification rule (MUST): Reject any adoption whose anchor_set.blob.jurisdiction != the settlement-active root that the verifier has configured/chosen for this store (Warrant §9). Only warrants reachable from that root (descendant closure) are eligible to authorize anything.

- Tool impact: anchor_governance.py should accept configured trust roots, compute reachability from the chosen root via prior edges, and compare anchor_set.jurisdiction to that root before considering adoption warrants. See P1‑A below.

Bottom line: with root pinning + embedded jurisdiction + settlement reachability, a fork cannot replay and be mistaken for the original. Without them, confusion attacks are feasible.

## 3) Attack the two-blob under split: profile blob + pure v0.3 threshold blob — can a satisfied threshold blob be replayed to authorize unrelated subjects; must the profile pin the threshold blob hash?

Yes, absent binding and cardinality rules, a satisfied threshold blob can be replayed across subjects in confusing ways. Two issues:

- Binding hole: The ADR text implies “changing either blob is constitutional change and must be adopted under the prior threshold,” but there is no machine-enforced binding between the profile and a specific threshold blob in the adoption path.
- Cardinality hole: The tool, as written, accepts any adoption whose under contains some profile blob and some v0.3 threshold that the envelope’s signatures satisfy; it does not require exactly one of each, nor that the threshold is the one the profile expects, nor that the profile is the current one adopted under the prior threshold.

P1 (spec silent; tool incomplete): Add explicit schema binding + adoption lineage + under cardinality; require verifiers to enforce them.

- Governance profile schema must pin the threshold:
  ```
  {
    "governance_profile": "sigma-glyph.anchor-governance@v1",
    "scope": "spec/ANCHORS.txt",
    "threshold": "<hex64: hash of the Warrant v0.3 threshold policy blob>"
  }
  ```
  - MUST: An anchor-set adoption warrant’s under array contains exactly one governance profile and exactly one threshold policy; the threshold hash MUST equal profile.threshold. Any extra governance or threshold blobs in under MUST make the adoption ineligible.

- Policy lineage (current-policy rule):
  - MUST: Governance policy changes (new profile P2 and its bound threshold T2) are adopted by a dedicated “profile adoption” warrant with subject.hash = H(P2), authorized under the threshold currently bound by the prior profile (P1.threshold = H(T1)). This imports Warrant §5.1 (“current policy controls its own replacement”) into governance.
  - MUST: The “current governance profile” is the unique settlement-active profile reachable from the configured root by DAG order with no conflicting maxima; anchor-set adoption MUST cite that current profile and its bound threshold.

- Verifier algorithm (deterministic; no clocks or git history):
  1) Select a jurisdiction root; compute settlement-active closure (Warrant §9).
  2) Walk profile-adoption warrants in that closure in DAG order; enforce that each was authorized under the prior bound threshold; derive the current profile P* and its threshold T*.
  3) For a candidate anchor adoption: require under = {P*, T*} exactly; require envelope signatures to satisfy T*; require anchor_set.blob.jurisdiction == chosen root.

Arithmetic demonstration (why it matters):
- Actors = {founder, A, B}, min_sigs = 2 (T).
- Malicious replay: envelope includes a doctored threshold T′ (e.g., min_sigs = 1) and gathers founder’s signature only.
- Naive tool: sees “some threshold present; signatures satisfy it” and accepts.
- Correct verifier: rejects because profile.threshold = H(T) ≠ H(T′); and absent a prior profile-adoption moving to T′ under T’s quorum, T′ is ineligible.

## 4) Actor deprecation mid-policy under Warrant §5.1 — concrete walk

Policy: threshold actors = {founder, model‑A, model‑B}, min_sigs = 2. model‑A retires silently (no warrants issued).

- Liveness immediately after A’s silent retirement: still live. Any two of three remain available; founder + model‑B can authorize anchor‑set adoptions under the unchanged threshold.

- Exact warrant sequence to recover/renew roster without founder ever becoming 1‑of‑1:
  1) Continue anchor-set adoptions under the current policy:
     - Adoption warrant: decision="accept", subject.hash=H(anchor_set vX+1), under=[P, T] where P.threshold=H(T).
     - Signatures: founder + model‑B (2/3).
  2) Rotate roster to reflect A’s retirement and add a successor C (preferred), or temporarily tighten to 2‑of‑2 (riskier):
     - Prepare T2:
       - Preferred: min_sigs=2, actors=[founder, model‑B, model‑C].
       - Temporary (if no successor yet): min_sigs=2, actors=[founder, model‑B] (note: brittle; any single absence halts).
     - Prepare P2 by copying P and updating "threshold": H(T2).
     - File a “profile adoption” warrant Wp2:
       - decision="accept", subject.hash=H(P2), under=[P, T].
       - Signatures: founder + model‑B (2/3 under T).
     - Future anchor-set adoptions MUST cite under=[P2, T2] and be signed by any two in the new actor set.

No step reduces to founder-alone authority. If both non‑founder actors retired simultaneously, liveness fails by design for min_sigs=2 unless a successor is available; the governance policy should be written to survive any single actor’s retirement (N≥3, min_sigs=2) and to stage successions before planned retirements.

## 5) Placement: Book III, a fourth Book, or a standing GOV document — argue one

Place ADR‑007 as a standing normative GOV document (outside the Books), anchored and versioned like the Books.

- Rationale:
  - Books I–III are protocol and federation. ADR‑007 governs the specification’s own release anchors and editorial succession — the project’s constitution — not on‑wire or on‑node behavior.
  - Folding governance into Book III would entangle jurisdiction mechanics with spec editorial policy and force implementations that never use spec governance to ship it.
  - A GOV document can normatively fix the anchor‑set schema, profile schema, and verifier algorithm (including the P1 fixes) and evolve independently without implying protocol changes.

Pointer hygiene:
- README should link GOV under “Specification Anchors and Governance.”
- The GOV doc must be governed by itself (bootstrap) and anchored in the repository’s ANCHORS.txt like the Books.

## Additional findings on ADR‑007 and tools (severity graded)

P1‑A: Settlement‑ignorant authorization in the tool (under‑checks Warrant v0.3 §9)
- Source reading: anchor_governance.py filters by local presence in .warrants/records and signature satisfaction but does not restrict to settlement‑active envelopes reachable from a configured root (Warrant §9).
- Risk: a non‑settlement envelope present in the store could be (wrongly) counted; multi‑root stores cannot be scoped.
- Fix (tool): add a trust config with explicit roots; compute reachable set from each chosen root via prior edges; ignore envelopes outside the closure; require a user‑selected root when multiple are configured; hard‑error if none configured.
- Spec (ADR‑007 “Mechanical decidability”): Verifiers MUST only consider adoption warrants that are settlement‑active in a configured jurisdiction per Warrant §9; envelopes not reachable from any trusted root MUST be ignored.

P1‑B: Under cardinality is unspecified; tool accepts multiples
- Source reading: tool tolerates multiple threshold/profile blobs in under and effectively picks one.
- Risk: ambiguous or adversarial under content yields divergent verifier decisions.
- Spec MUST: An anchor‑set adoption warrant’s under array MUST contain exactly one governance profile and exactly one threshold policy; more than one of either MUST make the adoption ineligible.

P1‑C: Ancestor discipline is only informal in ADR‑007
- Tool enforces “genesis MUST omit ancestor; successors MUST point to the immediate prior”; ADR text should promote this to a MUST with fork semantics.
- Spec MUST: Mismatch indicates a fork and MUST be rejected as an upgrade in this jurisdiction (present separately if intentionally a new jurisdiction with a different root).

P2‑A: If jurisdiction embedding (Section 2) is not adopted, at minimum document strict root pinning
- Document clearly that verifiers MUST pin a trusted root and ignore any adoption not in its descendant closure; explain expected first‑use pinning.

P2‑B: Tool dependency transparency
- Source reading: if cryptography/Ed25519 is unavailable, signatures cannot be verified and the tool reports “no satisfying adoption warrant,” which is ambiguous.
- Tool change: if Ed25519 unavailable, exit non‑zero with a clear message like “ERR: cannot verify signatures — install cryptography” unless an explicit “--no-sig” mode is chosen, which should always return “UNGOVERNED/NOT AUTHORIZED.”

P2‑C: Clarify “live sections” and completeness of anchors array
- The parser treats sections with parentheses as historical; live sections are evaluated oldest‑to‑newest.
- Spec SHOULD: State that at most one release section is live per line of descent; the anchors array MUST include every file declared live for the release (bundle completeness).

P2‑D: Canonical JSON (JCS/RFC 8785) parameters should be explicit
- Add an explicit note: canonical_json = RFC 8785 (JCS); name encoder parameters (sorted keys, no whitespace, UTF‑8).

P2‑E: Stores with key‑state warrants
- Tool currently refuses to derive key state; ADR‑007 should state that anchor governance in stores containing key‑state warrants requires a verifier that performs key‑state derivation (Warrant v0.3). anchor_governance.py is a scoped verifier and SHOULD refuse such stores with a clear message.

Editorial clarity (non‑blocking)
- Mechanical decidability: state that “authorized(anchor_set, store, trust_roots, bound_keys)” is a pure function; wall‑clock time, git history, or local filesystem timestamps MUST NOT enter the decision.
- Fork legitimacy: “A fork’s anchor‑sets MUST carry a different jurisdiction root; verifiers MUST present them as separate jurisdictions even if the anchor arrays coincide.”

Implementation sketch (for maintainers; not prescriptive)
- Enforce settlement activity (P1‑A) and jurisdiction field (Section 2).
- Enforce single profile/threshold and binding (Section 3).
- Discover current profile by walking profile‑adoption warrants in DAG order from the root; enforce “authorized under prior threshold.”
- Improve error visibility (P2‑B).
