## 0. Анотація та Архітектурний Принцип

Ця специфікація складається з двох нормативних Книг:

- **Книга I — Істина (The Book of Truth).** Побітово-детерміноване ядро: канонічні байти, хешування, редукція. Мінімальна, герметична, самодостатня. Вона визначає *що є* і *що означає обчислити*.
- **Книга II — Навігація (The Book of Navigation).** Хвильовий шар, координати, атрактори, інтерференція. Вона визначає *як орієнтуватися* серед того, що є.

**Головний закон розщеплення (MUST):**

> **Хвиля — це view, а не field.** WaveVectorQ НЕ входить у канонічні байти та НЕ впливає на NodeHash. Ідентичність визначається виключно структурою обчислення. Хвиля, фаза, колір, координати — анотації, адресовані *на* хеш, але не присутні *в* хеші.

Це усуває суперечність версії 0.2.12, де хвиля входила в хеш, але їй заборонялося впливати на семантику: два побітово ідентичні обчислення з різним "настроєм" були різними сутностями, що перетворювало semantic fork detection на шум. У 0.3.0 форк — це завжди форк *обчислення*, ніколи — форк настрою.

**Ключові слова** MUST, MUST NOT, SHOULD, MAY трактуються згідно з RFC 2119.

# КНИГА I — ІСТИНА (Bit-Exact Core)

## 1. Фундаментальні Структури (MUST)

### 1.1. Вузол (SigmaNodeV2)

```text
enum OpCode : uint8 {
  LITERAL    = 0x00,
  REF        = 0x01,
  APPLY      = 0x02,
  RESERVED   = 0x03,  // (колишній LAMBDA; див. §1.3)
  DISSONANCE = 0xFF
}

// Flags: F_ATOM = 0x01, F_LEFT = 0x02, F_RIGHT = 0x04
```

**Інваріанти (MUST):**

* Будь-які біти в `Flags` поза маскою `0x07` MUST be zero.

| OpCode     | Flags (MUST equal)   | Semantics                              |
| ---------- | -------------------- | -------------------------------------- |
| LITERAL    | `F_ATOM`             | `atom = SHA-256(DataBLOB)`             |
| REF        | `F_ATOM`             | `atom = TargetHash` (indirection)      |
| APPLY      | `F_LEFT \| F_RIGHT`  | `left = Fn`, `right = Arg`             |
| DISSONANCE | `F_ATOM`             | `atom = ReasonHash`                    |

**LITERAL — інертний commitment (MUST):** канонічний вузол містить лише digest `SHA-256(DataBLOB)`, не сам blob. Для редукції blob **ніколи не потрібен**: LITERAL — нормальна форма, розпізнавання комбінаторів іде за NodeHash, а не за вмістом. BlobStore — окремий контракт поза Книгою I: `blob(atom)` MAY бути недоступним без впливу на обчислення; при отриманні blob валідація `SHA-256(blob) == atom` є MUST; `max_blob_size` — локальний resource guard (§3.6).

### 1.2. Канонічна Серіалізація (MUST)

* **Layout:** `[Op:1][Flags:1][Atom?:32][Left?:32][Right?:32]`
* **Order:** опціональні поля MUST з'являтися в порядку: `Atom`, потім `Left`, потім `Right`.
* **Field mapping:** `F_ATOM → Atom`, `F_LEFT → Left`, `F_RIGHT → Right`. Інших полів для SigmaNodeV2 не існує.
* **Hash:** `NodeHash = SHA-256(CanonicalBytes)`.
* Внутрішнє представлення хешу — 32 raw bytes. Hex — лише presentation.

Зауваження: у канонічних байтах **немає** WaveVectorQ. Мінімальний вузол — 2 байти + поля. Це і є фізичне втілення Головного закону.

### 1.3. Доля LAMBDA (MUST)

Опкод `0x03` зарезервовано і MUST NOT використовуватись у канонічних байтах SigmaNodeV2. Обчислювальний базис Книги I — комбінатори **S, K, I** (див. §3): вони повні за Тюрингом і не мають змінних, тому проблеми зв'язування (alpha-еквівалентність, de Bruijn) не вирішуються — вони **не виникають**. Терми зі змінними MAY існувати на рівні інструментів і MUST компілюватися в SKI (bracket abstraction, Appendix C) *до* серіалізації та хешування.

## 2. Genesis (Канонічні Атоми)

### 2.1. Трійця — номінальні аксіоми (MUST)

Комбінатори визначаються **номінально**: кожен — це LITERAL, чий атом є SHA-256 ASCII-імені. Це аксіоми: вони не мають внутрішньої структури, їхній зміст задається правилами редукції (§3).

| Glyph | Node CanonicalBytes (hex)              | NodeHash                                                           |
| ----- | -------------------------------------- | ------------------------------------------------------------------ |
| **I** | `0001` + SHA-256("I")                  | `2f33694d09810641fa5b8c47a7c0dc42e1b99eb8c9784a00aaee9a66330f4162` |
| **K** | `0001` + SHA-256("K")                  | `bc0c2fe26e44e2aed8ce500a74963bc270fd4a49ec0c2e4837ce7a64bb0a486c` |
| **S** | `0001` + SHA-256("S")                  | `887045bc22935aec5cba2dc11400d4e4357bc34d06681a6e92f06e7795b1f8a6` |

Довідково: SHA-256("I") = `a83dd0ccbffe39d071cc317ddf6e97f5c6b1c87af91919271f9fa140b0508c6c`; SHA-256("K") = `86be9a55762d316a3026c2836d044f5fc76e34da10e1b45feee5f18be7edb177`; SHA-256("S") = `8de0b3c47f112c59745f717a626932264c422a7563954872e237b223af4ad643`.

**Розпізнавання комбінатора (MUST):** вузол *є* комбінатором I/K/S тоді й лише тоді, коли його NodeHash дорівнює відповідній канонічній константі. Identity by Hash — навіть у семантиці.

### 2.2. FALSE — перша теорема (MUST)

На відміну від 0.2.12, де FALSE був номінальним (`LITERAL "FALSE"`), у 0.3.0 FALSE — **структурний**:

```text
FALSE ≡ APPLY(K, I)
CanonicalBytes = 0206 || NodeHash(K) || NodeHash(I)
NodeHash(FALSE) = 65cd957fee7ec9fb310bc9d9712cec1726c78f8026fda679ac8f237938a32098
```

Екстенсіонально `(K I) a b →* b` — селектор другого аргумента, класичний Church-FALSE. Трійця — аксіоми (номінальні), все інше — теореми (структурні). Це нормативний зразок для всіх подальших похідних артефактів.

## 3. Нормативна Семантика Редукції (MUST)

Це розділ, якого не існувало в 0.2.12: серіалізація без операційної семантики описувала, як зберігати терм, але не що означає його виконати.

### 3.1. Правила переписування (SKI Graph Rewriting)

Позначення: `⟨X⟩` — вузол, чий NodeHash дорівнює канонічному хешу X.

```text
R-I:  APPLY(⟨I⟩, x)                     →  x
R-K:  APPLY(APPLY(⟨K⟩, x), y)           →  x
R-S:  APPLY(APPLY(APPLY(⟨S⟩,x),y), z)   →  APPLY(APPLY(x, z), APPLY(y, z))
R-R:  REF(h)                            →  resolve(h)        // indirection
```

### 3.2. Порядок редукції (MUST)

**Normal order (leftmost-outermost).** На кожному кроці редукується єдиний leftmost-outermost редекс. Це робить послідовність кроків — і отже витрату ATP — детермінованою функцією терма.

Формально, для терма `t`:

```text
step(t):
  if t = REF(h):                            fire R-R
  elif t matches R-I | R-K | R-S at root:   fire that rule
  elif t = APPLY(f, a):
      if step(f) exists: reduce inside f
      elif step(a) exists: reduce inside a
      else: none            // t is in normal form
  else: none
```

### 3.3. ATP — паливо (MUST)

* Кожне спрацювання правила (R-I, R-K, R-S, R-R) коштує рівно **1 ATP**.
* `eval(term, atp_budget: uint32)` виконує кроки, доки: (a) досягнуто нормальної форми → результат = терм; (b) бюджет вичерпано до досягнення нормальної форми → результат = `DISSONANCE(ATP Exhausted)`; (c) R-R не може розв'язати хеш → результат = `DISSONANCE(Unresolved Reference)`.
* Нормативна модель — **переписування термів (tree semantics)**. Імплементації MAY використовувати graph reduction зі спільними підтермами, але спостережуваний результат і облік ATP MUST збігатися з tree semantics.
* Результат `eval` — це вузол; його NodeHash — канонічна адреса результату. Таким чином виконується контракт `result_hash = eval(term_hash, atp)` — детермінований, без флоатів, адресований контентом.

### 3.5. Resolution Contract (MUST)

`resolve(h)` — єдина операція отримання вузла з CAS/store за хешем. Вона використовується: (a) для кореневого терма, (b) правилом R-R, (c) для матеріалізації дітей `APPLY` під час пошуку редекса. **Будь-яка** невдача `resolve(h)` — кореня, REF-цілі чи дитини APPLY — MUST давати канонічний результат `DISSONANCE(Unresolved Reference)`. Байти в store, що не проходять валідацію §4.1, матеріалізуються як Canonical Invalid Object (отруєний вузол видно, але він не ламає детермінізм).

### 3.6. Resource Faults ≠ Dissonance (MUST)

Розрізняються два класи відмов:

* **Канонічні** (детерміновані, частина семантики, однакові на всіх нодах): нормальна форма, `DISSONANCE(ATP Exhausted)`, `DISSONANCE(Unresolved Reference)`.
* **Локальні resource faults** (залежать від заліза/конфігурації ноди): перевищення `max_node_depth`, `max_materialized_nodes`, `max_store_fetches`, `max_blob_size`, `max_result_size`. Порушення MUST підніматися як **implementation fault** (відмова виконання), і MUST NOT серіалізуватися як DISSONANCE — інакше локальний ліміт однієї ноди отруїв би канонічний простір результатів. Мотивація лімітів: R-S подвоює підтерм за 1 ATP, тож розмір може рости як O(2^ATP) — ATP обмежує *роботу*, але не *пам'ять*; пам'ять обмежують локальні guards.

## 4. Валідація (MUST)

### 4.1. Десеріалізація

1. `len(buffer) MUST be >= 2`.
2. Read `[Op:1][Flags:1]`.
3. Validate `Flags & ~0x07 == 0` та відповідність таблиці §1.1 (точна рівність).
4. `expected_len = 2 + 32 * popcount(Flags & 0x07)`; `popcount` рахує встановлені біти unsigned.
5. `len(buffer) MUST equal expected_len`.
6. `Op == 0x03 (RESERVED)` → невалідний.
7. Будь-яка помилка → Canonical Invalid Object (§4.2).

### 4.2. Canonical Invalid Object (MUST)

```text
Op=DISSONANCE, Flags=F_ATOM, Atom=SHA-256("Invalid Object")
CanonicalBytes: ff017cc62bcc7c921683532cec1c1c331ca81d76b001e0c7f407a4078df7f696efe8
NodeHash:       af69b5176c7ac3855c2eac3d1f6159c74d5328e92aac0a33cdba68bbaeba4507
```

## 5. Test Vectors (MUST PASS)

Усі вектори нижче обчислені та перевірені референсною реалізацією.

**TV2-1 (LITERAL I):**
Bytes: `0001a83dd0ccbffe39d071cc317ddf6e97f5c6b1c87af91919271f9fa140b0508c6c`
Hash: `2f33694d09810641fa5b8c47a7c0dc42e1b99eb8c9784a00aaee9a66330f4162`

**TV2-2 (APPLY, FALSE = K·I):**
Bytes: `0206` + NodeHash(K) + NodeHash(I)
Hash: `65cd957fee7ec9fb310bc9d9712cec1726c78f8026fda679ac8f237938a32098`

**TV2-3 (DISSONANCE, ATP Exhausted):**
Bytes: `ff01dc435a08513893bacd07abd802b9c526e92ae57ca6db40c1c8f369fd7032e090`
Hash: `8bb0006f4c0a51a645877c10db80b7360b0d34f6f826e5737d0847f8b1493176`

**TV2-4 (Reduction, один крок):**
`T = APPLY(⟨I⟩, ⟨K⟩)`, hash `51d8148feda28f17304c9ed6c34d9d548c83a84c380f4dd1ba0a037ceb9d4d3e`.
`eval(T, 1) = ⟨K⟩` (hash `bc0c2fe2…`), спожито 1 ATP. `eval(T, 0) = DISSONANCE(ATP Exhausted)`.

**TV2-5 (Reduction, теорема SKK=I):**
`T = APPLY(APPLY(APPLY(⟨S⟩,⟨K⟩),⟨K⟩), ⟨I⟩)`, hash `c9f57b3f594d7b72b0855b0d6fabba89e6ccdf6840c8f84aeb5fd4707300bbfc`.
Крок 1 (R-S): → `APPLY(⟨FALSE⟩, ⟨FALSE⟩)` — проміжний терм має hash `b45355fcf51a189b7ec00a73870d9bf4b8c6e830419ba1214eecd7dc4eaba133`. (Проміжна форма редукції SKK — це буквально FALSE, застосований до FALSE; система жартує сама з себе.)
Крок 2 (R-K): → `⟨I⟩`.
`eval(T, 2) = ⟨I⟩`, спожито 2 ATP. `eval(T, 1) = DISSONANCE(ATP Exhausted)`.

**TV2-6 (Duplication Stress — нормативність tree semantics):**
`T = APPLY(APPLY(APPLY(⟨S⟩,⟨I⟩),⟨I⟩), APPLY(⟨I⟩,⟨K⟩))`, hash `0379bafee726f493bffc153163b7165b916efe0bd661cf99bc2f834f36db8198`.
Нормальна форма: `APPLY(⟨K⟩,⟨K⟩)`; спожито рівно **5 ATP** (tree semantics: дубльований R-S редекс `I·K` редукується двічі). Graph reduction з шарингом виконав би це за 4 кроки — реалізація з шарингом MUST усе одно звітувати 5. Цей вектор — конформанс-тест проти "економії" ATP оптимізаторами.

**TV2-7 (Omega — детермінована незупинність):**
`Ω = APPLY(W, W)`, де `W = APPLY(APPLY(⟨S⟩,⟨I⟩),⟨I⟩)`; hash `0609d7e3bac2c6927c34ade51c7d6728a75c6ac0206fdb184524843b4fb94211`.
Для будь-якого бюджету `n`: `eval(Ω, n) = DISSONANCE(ATP Exhausted)`. ATP — гарантія тотальності.

**TV2-8 (Unresolved Child):**
`APPLY(⟨I⟩, ghost)`, де `ghost` відсутній у store → `DISSONANCE(Unresolved Reference)` (контракт §3.5 покриває дітей APPLY, не лише REF).

**Негативні (RECOMMENDED):** flags поза `0x07`; `Op=LITERAL, Flags≠F_ATOM`; `Op=0x03`; `len ≠ expected_len` — усе → Canonical Invalid Object.

*Кінець Книги I. Все, що нижче, MUST NOT впливати на канонічні байти, хешування чи редукцію.*

# КНИГА II — НАВІГАЦІЯ (Wave / Coordinate Layer)

## 6. Scope Guard (MUST)

* Книга II — інтерпретаційний шар поверх Книги I.
* MUST NOT: ніщо в Книзі II не змінює редукцію, серіалізацію чи хешування.
* MUST: у разі видимого конфлікту Книга I має пріоритет.
* **Axiom (успадковано з E.0.1):** Phase is not an identifier. Identity is defined ONLY by the full NodeHash. Фаза — координата навігації; кілька об'єктів в одному секторі — density cluster (Mass).

## 7. Хвильова Анотація (WaveVectorQ as View)

### 7.1. Структура

```text
struct WaveVectorQ { ph: uint16; am: uint16; en: int16; }   // Big-Endian
```

### 7.2. Адресація (MUST)

Хвиля зберігається **окремо** від вузла і адресується його хешем:

```text
WaveAnnotation = (NodeHash: 32 bytes) → WaveVectorQ
```

Один вузол MAY мати різні хвильові анотації в різних контекстах/нодах федерації — це *не форк*: обчислення ідентичне, розходиться лише навігаційний настрій. Розбіжність анотацій — сигнал для гередотичного (gossip) узгодження, не для семантичної тривоги.

### 7.3. Конвенції за замовчуванням (SHOULD)

* Genesis-атоми: `{Ph_канонічна, 65535, -32768}` (див. §9).
* DISSONANCE: `{0, 0, 0}` — мертва хвиля (успадковує інваріант 0.2.12, тепер як конвенція view-шару, а не закон ідентичності).

### 7.4. Пріоритет анотацій: Pin > Derived (MUST)

Хвильова анотація вузла походить з одного з двох джерел:

1. **Pin (аксіоматична координата):** явно призначена в канонічних таблицях Книги II (Трійця §9.2, Grand Cross §9.3, Time Anchor §9.4, Пантеон §9.5).
2. **Derived (похідна):** обчислена інтерференцією `wave(APPLY(f,a)) = interfere(wave(f), wave(a))`.

Pin MUST мати пріоритет над Derived. Це знімає видимий конфлікт: за Законом Лівого Домінування `APPLY(K,I)` успадкував би фазу K (180°), але FALSE має **Pin 270°** — кардинальна точка призначена аксіоматично, Kwen Rule (§9.3) пояснює *чому* саме так, інтерференція тут не застосовується. Kwen Rule підвищено до **SHOULD** для навігаційного планування K-type застосувань; для канонічних байтів та NodeHash вона, як і раніше, MUST NOT.

## 8. Інтерференція (Bit-Exact у межах Книги II)

Хвильова математика лишається побітово детермінованою — вона нормативна для *навігаційного* шару.

### 8.1. Fixed-Point примітиви (MUST)

```text
def div_round_half_up(n, d):
    # round-half-away-from-zero; d MUST be > 0
    # n MUST бути промотований до ширшого знакового типу перед негацією
    s = -1 if n < 0 else 1
    a = -n if n < 0 else n
    q = a // d
    r = a % d
    if 2*r >= d: q += 1
    return s * q

def clamp_i16(x): return -32768 if x < -32768 else (32767 if x > 32767 else x)
```

### 8.2. LUT — тепер пришпилена (MUST)

* **Range:** `[-32767..32767]`; `-32768` заборонене (формула його не породжує).

### 8.4. Закон Лівого Домінування (MUST) — The Law of Left Dominance

`new_ph = w1.ph`: фаза результату — фаза **лівого** операнда. Амплітуда та ентропія симетричні; фаза — ні:

```text
interfere(A, B).ph = A.ph  ≠  interfere(B, A).ph = B.ph
```

**Це навмисно і нормативно.** Обґрунтування: інтерференція моделює *застосування* — а застосування некомутативне за своєю природою: `f(x) ≠ x(f)`. Функція визначає напрямок (фазу) результату; аргумент модулює його силу (амплітуду) та впорядкованість (ентропію). Аналогія: кватерніонний добуток некомутативний саме тому, що композиція обертань залежить від порядку — навігаційний простір Σ-GLYPH успадковує цю властивість. Симетризація фази (напр. усереднення) заборонена: вона знищила б інформацію про те, *хто діяв*, і зробила б навігацію дезорієнтованою в графах APPLY.
